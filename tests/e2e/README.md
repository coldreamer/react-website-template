# 🔥 ConversionForm E2E 测试文档

> 由一名专业的"毒舌 QA"编写的端到端测试套件

## 📋 测试概述

本测试套件使用 **Playwright** 对 `ConversionForm` 组件进行全面的端到端测试，确保表单在各种场景下都能正常工作。

## 🎯 测试用例

### 1. 非法邮箱验证测试 ✅

**测试目标**：验证表单是否能正确拦截并显示非法邮箱的错误提示

**测试场景**：

- ❌ `notanemail` - 缺少 @ 符号
- ❌ `missing@domain` - 缺少域名后缀
- ❌ `@nodomain.com` - 缺少用户名
- ❌ `spaces in@email.com` - 包含空格
- ❌ `double@@domain.com` - 双 @ 符号
- ❌ `trailing.dot.@test.com` - 结尾有点
- ❌ 空邮箱

**预期结果**：

- 所有非法邮箱都应该被拦截
- 显示错误提示："请输入有效的邮箱地址"
- 表单不应该被提交

### 2. Loading 状态测试 ✅

**测试目标**：验证在网络延迟（3秒）时，Loading 状态是否正常触发

**测试场景**：

- 模拟 3 秒网络延迟
- 提交表单
- 检查 Loading 状态

**预期结果**：

- ✅ 提交按钮立即被禁用（500ms 内）
- ✅ 显示 Spinner 加载动画（500ms 内）
- ✅ 按钮文字变为"提交中..."（500ms 内）
- ✅ Loading 状态持续至少 2 秒
- ✅ 在 Loading 期间，按钮保持禁用状态

### 3. 成功 UI 测试 ✅

**测试目标**：验证提交成功后是否显示优雅的成功页面

**测试场景**：

- 填写完整表单
- 提交表单
- 等待成功响应

**预期结果**：

- ✅ 显示成功标题："提交成功！"
- ✅ 显示成功描述文字
- ✅ 显示绿色的成功图标
- ✅ 显示"返回表单"按钮
- ✅ 点击"返回表单"后回到表单页面
- ✅ 表单字段被重置为空

### 4. 响应式设计测试 ✅

**测试目标**：验证移动端显示是否正常

**测试场景**：

- 设置移动端视口（375x667）
- 检查表单显示

**预期结果**：

- ✅ 表单在移动端正确显示
- ✅ 表单占满屏幕宽度（带适当 padding）

### 5. 边界条件测试 ✅

**测试目标**：验证所有字段的边界条件

**测试场景**：

- 姓名太短（1 个字符）
- 姓名太长（51 个字符）
- 手机号太短（5 位）
- 手机号包含非数字字符

**预期结果**：

- ✅ 显示对应的错误提示
- ✅ 表单不应该被提交

## 🚀 运行测试

### 安装依赖

```bash
npm install
```

### 运行所有测试

```bash
npm run test:e2e
```

### 使用 UI 模式运行（推荐）

```bash
npm run test:e2e:ui
```

这会打开 Playwright 的 UI 界面，可以：

- 查看测试执行过程
- 调试失败的测试
- 查看测试报告
- 重新运行单个测试

### 运行测试（显示浏览器）

```bash
npm run test:e2e:headed
```

### 调试模式

```bash
npm run test:e2e:debug
```

### 查看测试报告

```bash
npm run test:e2e:report
```

## 🎭 测试配置

测试配置文件：`playwright.config.ts`

主要配置：

- **测试目录**：`./tests/e2e`
- **超时时间**：30 秒
- **基础 URL**：`http://localhost:3000`
- **浏览器**：Chromium（桌面）+ iPhone 13（移动）
- **截图**：失败时自动截图
- **视频**：失败时保留录像
- **自动启动开发服务器**：是

## 🛠️ 测试辅助工具

`tests/e2e/helpers.ts` 提供了一些实用的辅助函数：

- `fillCompleteForm()` - 快速填写完整表单
- `expectToast()` - 等待并验证 Toast 消息
- `waitForLoading()` - 等待 Loading 状态
- `waitForSuccessPage()` - 等待成功页面
- `mockNetworkDelay()` - 模拟网络延迟
- `takeScreenshot()` - 截图辅助函数

1. **随机失败**：由于表单有 20% 的模拟失败率，成功 UI 测试可能需要重试。测试代码已包含自动重试逻辑。
